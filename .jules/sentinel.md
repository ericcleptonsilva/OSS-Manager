## 2024-05-05 - [Critical] Hardcoded Fallback Credentials & Admin Role Bypasses
**Vulnerability:** The application contained severe privilege escalation flaws. The default user role initialized to `admin`, fallbacks for unknown or standard users without roles defaulted to `admin`, and specific email addresses (`ericlobobr.01@gmail.com`, `admin@oss.com`) had hardcoded backdoors in `App.tsx` granting them admin rights regardless of their actual database state. Additionally, `services/parseService.ts` permitted custom logins using hardcoded fallback credentials (`admin@oss.com` / `admin`) if the database query succeeded but the fields were potentially manipulated or empty.
**Learning:** Hardcoding credentials and default privilege assignments directly in frontend components (`App.tsx`) bypasses all backend security and authorization logic. Relying on default fallbacks to the highest privilege level (`admin`) instead of the least privilege (`student`) ensures that any failure in authorization verification results in full system compromise. Furthermore, failing to strictly type-check and validate the non-emptiness of database fields before comparing them with user input in custom login flows can lead to authentication bypasses (e.g., if a user submits an empty password and the database field is also unexpectedly empty or undefined).
**Prevention:**
1.  **Least Privilege Defaults:** Always default user roles and unauthenticated states to the lowest possible privilege level (e.g., `student` or `guest`).
2.  **No Hardcoded Backdoors:** Never hardcode email addresses, usernames, or passwords in frontend code to grant special access. All authorization must be derived from a secure, verified backend source.
3.  **Strict Credential Validation:** When implementing custom authentication flows, explicitly verify that retrieved credentials (like `storedAdminEmail` and `storedAdminPass`) are strings, are not empty (`trim() !== ""`), and exactly match the user input using strict equality (`===`) before granting access.
