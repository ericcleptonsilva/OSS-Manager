## 2024-05-09 - Admin Override Privilege Escalation
**Vulnerability:** A hardcoded bypass in frontend and backend logic granted full admin privileges solely based on checking matching string emails (e.g., 'admin@oss.com' and 'ericlobobr.01@gmail.com'), allowing potential privilege escalation without proper cryptographic authentication checks, as well as an empty credentials bypass. Furthermore, an absent role fallback granted 'admin' privileges to any user by default.
**Learning:** Hardcoding 'override' administration access directly into the client-side (`App.tsx`) and bypassing actual authentication allows users who can mock the email input or lack a defined role to instantly gain full system access.
**Prevention:** Role assignment (`setUserRole`) must strictly rely on secure role checks returned from the backend (or fallback to 'student' - least privilege). Fallback authentication handlers (`performCustomLogin`) must explicitly reject empty strings to prevent bypasses.
