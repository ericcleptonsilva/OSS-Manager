## 2024-04-27 - Hardcoded Backdoor Credentials and Empty String Auth Bypass
**Vulnerability:** The `performCustomLogin` authentication fallback contained hardcoded credentials (`admin@oss.com` and `admin`) that granted backdoor admin access. Additionally, there was no validation for empty string credentials, meaning an attacker could bypass authentication or cause unexpected behavior by submitting empty emails or passwords. The `App.tsx` also hardcoded logic to force an admin role for `ericlobobr.01@gmail.com` and `admin@oss.com` regardless of the actual Parse User role, bypassing authorization logic, and the UI artificially forced the team's admin email back to `admin@oss.com` when editing the team settings.
**Learning:** Hardcoded credentials and explicit email checks for role assignment bypass standard authentication and authorization controls, creating backdoors that cannot be disabled without code changes. Fallback authentication mechanisms that directly query string fields must strictly validate inputs against empty strings, as `String.trim()` or partial checks can result in matching empty or generic fallback fields (like `"" === ""`).
**Prevention:** Remove all hardcoded credentials and email-based role overrides. Rely exclusively on robust Role-Based Access Control (RBAC) provided by the authentication provider (e.g., Parse Server `Role` objects). Ensure that custom authentication logic strictly rejects empty inputs and verifies that retrieved fallback credentials from the database are not null/empty before performing comparisons. Never hardcode logic in UI components that resets user-configured security fields.
