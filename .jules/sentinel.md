## 2024-05-18 - [Privilege Escalation & Hardcoded Credentials]
**Vulnerability:** The application previously relied on hardcoded admin emails (`admin@oss.com`, `ericlobobr.01@gmail.com`) for overriding user roles to `admin` in `App.tsx` and also defined fallback default credentials (`admin@oss.com` / `admin`) that could potentially be matched against empty strings during custom login loops if fields were wiped.
**Learning:** Hardcoded overrides and default backdoor fallback scenarios represent severe privilege escalation risks. Specifically, providing explicit "known emails" or relying on default constants bypasses dynamic authentication state entirely. Additionally, custom authentication loops matching against default variables are risky if those default variables are susceptible to being evaluating equivalently to empty client inputs.
**Prevention:** Hardcoded credential overloads and backdoors should be entirely avoided. Always follow the principle of least privilege - defaulting fallback roles in the UI specifically to standard access limits (e.g. `student`) instead of highest (e.g. `admin`). When checking explicit roles or permissions, never trust a single string email list; use established Auth token strategies or dedicated user schemas. Furthermore, always sanitize inputs and perform strict undefined/empty checks before credential matching algorithms.
