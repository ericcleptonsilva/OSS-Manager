## 2024-05-24 - Hardcoded Authentication Backdoors
**Vulnerability:** The application contained hardcoded backdoor credentials (`admin@oss.com` and `ericlobobr.01@gmail.com`) that automatically granted administrator privileges on the frontend merely by typing the email during login. Furthermore, the legacy custom login fallback in `parseService.ts` would authenticate any user attempting to log in with `admin@oss.com` and the password `admin`.
**Learning:** Hardcoded overrides intended for testing or initial administrative access were accidentally left in the production source code. These created severe security bypasses because authentication logic in `App.tsx` relied entirely on the presence of a specific email string without requiring validation or password authentication, completely bypassing RBAC and identity verification. Additionally, the backend service lacked robust checks, meaning empty database fields could potentially match an empty inputted password.
**Prevention:** Never leave testing backdoors or "admin overrides" in production code. All roles and administrative access must be managed natively via Parse RBAC (Roles/Permissions) and verified strictly via secure token-based logins (e.g., `Parse.User.current()`), not client-side email string matching. Empty-string validation must be explicitly implemented when querying custom credential fallback tables.
