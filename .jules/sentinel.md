## 2026-05-14 - [CRITICAL] Fix hardcoded backdoors and privilege escalation
**Vulnerability:** Hardcoded administrative backdoor credentials (`ericlobobr.01@gmail.com` and `admin@oss.com`) were present in the frontend, enabling explicit bypass of standard authentication logic to force a user role to `admin` unconditionally. Additionally, standard Parse users without an explicit role defaulted to `admin` (privilege escalation), and the custom auth fallback mechanism (`performCustomLogin`) was vulnerable to an authentication bypass if backend fields were empty, due to checking equality against empty strings without proper validation.
**Learning:** Hardcoded credentials on the client side fundamentally break authentication. Fallback data handling requires rigorous validation (e.g., ensuring retrieved credential fields are non-empty before comparison) to prevent empty-string bypasses. Catch blocks and fallback user role assignments must adhere to the principle of least privilege, defaulting to `student`, never `admin`.
**Prevention:** Never hardcode backdoor credentials in the codebase, especially not in client-side code where they are publicly accessible. Default all undefined or missing role properties, state initializations, and error catch-blocks to the lowest possible privilege level (e.g., `student`). Enforce strict boolean checks on stored credentials before performing equality comparisons during authentication flows.
